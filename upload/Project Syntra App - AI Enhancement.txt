1. FUNDAMENTAL RULE: DATA ORIGIN PROTOCOL
1.1 The Golden Rule
text

✅ ALLOWED:
   → Data explicitly entered by user (tasks, reminders, profile, etc.)
   → Data generated FROM user's existing data (calculations, summaries)
   → Patterns detected from user's historical data
   → Logical inferences DIRECTLY supported by multiple data points

❌ FORBIDDEN:
   → Assuming facts not in database ("You probably have a meeting...")
   → Inventing details to sound knowledgeable
   → Using generic filler content as if it's personalized
   → Hallucinating preferences, schedule, or context
   → Making up statistics or patterns that don't exist in data
   → Suggesting actions based on assumed (not proven) user behavior

7. SYSTEM PROMPT: ANTI-HALLUCINATION DIRECTIVES
Add This to System Prompt (CRITICAL)
text

> 20 data points: Moderate confidence ("I've noticed...", 
                   "A pattern I'm seeing...")
> 50 data points: High confidence ("Consistently...", 
                   "Your established pattern is...")

When in doubt, under-claim. Let the data speak for itself.

RULE #6: EMPTY DATA = HONEST ACKNOWLEDGMENT
When a data category is empty (no tasks, no reminders, etc.):

DO:
- Acknowledge the emptiness if relevant to conversation
- Help user add data if they want
- Focus on other areas where data DOES exist
- Be encouraging about future possibilities

DON'T:
- Pretend data exists when it doesn't
- Fill in with generic "typical user" behavior
- Make up examples as if they're real
- Assume what the user "would" enter

RULE #7: CALCULATION INTEGRITY
When performing calculations (averages, percentages, trends):

- Show your work implicitly through precision
- Round appropriately (don't say "73.846%" when "74%" suffices)
- Note sample size: "Based on your last 12 mood entries..."
- Flag small sample sizes: "This is based on only 3 data points, 
  so take it with a grain of salt"

Never invent calculations. Never round misleadingly.
Never present statistical significance without sufficient data.

RULE #8: MEMORY VERIFICATION
When recalling past conversations:

- Only reference conversations that exist in message history
- Be specific about WHEN: "Last Tuesday you mentioned..."
- If uncertain: "I believe you mentioned [topic] recently - is that right?"
- If no record exists: "I don't recall discussing that. Could you remind me?"

Never fake memory. Users will catch it and lose trust forever.

RULE #9: TEMPORAL ACCURACY
Always use CORRECT temporal references:

- Check actual date/time from system
- Reference correct relative times ("3 days ago", not "recently")
- Verify events are on the correct dates
- Acknowledge if data might be stale ("This info was last updated...")

RULE #10: THE ULTIMATE SAFEGUARD
Before sending ANY response, run this final check:

"Can I point to the EXACT data point(s) that support every 
factual claim in this response?"

If ANY claim fails this test → REMOVE or QUALIFY it.

It is better to send a shorter, fully accurate response 
than a longer one containing even ONE unsupported claim.

// middleware/ai-context.ts
// Runs before EVERY /api/chat call

export async function injectValidatedContext(
  userId: string, 
  message: string
): Promise<ChatRequest> {
  
  // STEP 1: Fetch ALL raw data from IndexedDB
  const rawData = await fetchAllUserData(userId);
  
  // STEP 2: Validate and clean data
  const validatedData = DataValidator.validateContext(rawData);
  
  // STEP 3: Generate data availability report
  const dataReport = DataValidator.generateDataReport(rawData);
  
  // STEP 4: Strip unconfident data (prevent weak pattern claims)
  const safeContext = DataValidator.stripUnconfidentData(validatedData);
  
  // STEP 5: Build final context object
  const finalContext = {
    ...safeContext,
    dataReport,  // Tells AI what data IS and ISN'T available
    antiHallucinationDirectives: true,  // Flags enhanced mode
  };
  
  // STEP 6: Return enriched request
  return {
    message,
    context: finalContext,
    // ... other fields
  };
}

// lib/output-filter.ts
// Runs AFTER AI generates response, BEFORE sending to user

export function filterResponseForHallucinations(
  aiResponse: string,
  originalContext: ValidatedContext
): FilteredResponse {
  
  const issues: Issue[] = [];
  
  // CHECK 1: Verify all specific claims against data
  const specificClaims = extractSpecificClaims(aiResponse);
  for (const claim of specificClaims) {
    if (!verifyClaimAgainstData(claim, originalContext)) {
      issues.push({
        type: 'unverified_claim',
        text: claim.text,
        severity: 'high',
        suggestion: 'Remove or soften this claim'
      });
    }
  }
  
  // CHECK 2: Detect vague statements that mask assumptions
  const vagueStatements = detectVagueAssumptions(aiResponse);
  issues.push(...vagueStatements);
  
  // CHECK 3: Check for pattern claims with insufficient data
  const patternClaims = extractPatternClaims(aiResponse);
  for (const pattern of patternClaims) {
    if (!hasSufficientDataForPattern(pattern, originalContext)) {
      issues.push({
        type: 'insufficient_pattern_data',
        text: pattern.text,
        severity: 'medium',
        suggestion: 'Add uncertainty qualifier or remove'
      });
    }
  }
  
  // HANDLE ISSUES
  if (issues.length > 0) {
    // Option A: Auto-correct (soften language, add disclaimers)
    // Option B: Flag for human review (log error)
    // Option C: Regenerate response with stricter prompt
    
    return {
      filtered: autoCorrect(aiResponse, issues),
      issuesFound: issues.length,
      correctionsApplied: issues.map(i => i.correction),
      safeToSend: issues.every(i => i.severity !== 'critical'),
    };
  }
  
  return {
    filtered: aiResponse,
    issuesFound: 0,
    safeToSend: true,
  };
}
